# Changelog

All notable changes to pilar are documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/), and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html). Per the project's load-bearing decision #5 (recorded in [`IMPLEMENTATION_ROADMAP.md`](./IMPLEMENTATION_ROADMAP.md)), explicit `version` is set in `.claude-plugin/plugin.json` only on tagged releases; the field is omitted between releases so the commit SHA serves as the version during active development.

## [Unreleased]

Phases 4, 5, 6, and 7 closed: sprint engine + Primary Collaborator core (P4); Editor and Fact-Checker subagents with the Independence Contract codified for real evaluation (P5); KB Librarian with intake, gap detection, and aspirational-statement registration (P6); per-pillar development with scaffolding + narrative + statements + status state machine (P7). ### P7 refinement

- **`/pilar:explore <pillar-id> [--angle "<hypothesis>"]`** ‚Äî bounded, hypothesis-driven per-pillar exploration of KB source content. The writer brings a one-sentence angle; the command ranks manifest REFs by likely relevance, lets the user pick which to dig into, and uses the Read tool on the picked source files (PDFs included via the Read tool's `pages` parameter ‚Äî defaulting to first 10 pages with user opt-in for deeper page ranges). Surfaces key passages and concepts; produces a 1‚Äì2 paragraph synthesis covering what the exploration supports, refutes, or suggests. Discarded by default; on save, appends a dated H2 section to `explorations/<pillar_id>.md` (one file per pillar; sections accumulate across invocations). Closes the original P7 gap where `/pilar:pillar-narrative` reads manifest entries grouped by `type` and `/pilar:pillar-statements` surfaces REF metadata, but neither command opens source files. (commands/explore.md)
- **`/pilar:init` extended** ‚Äî `explorations/` directory added to the ¬ß3 directory tree alongside `pillars/`, `registers/`, `sprints/`, `qc/`, `knowledge-base/`. Holds optional per-pillar exploration notes; the directory is committed (with `.gitkeep`) but its files are scratch markdown ‚Äî not ¬ß7 artifacts and not schema-validated. (commands/init.md)
- **`examples/p7-fixtures/explorations/p-04.md`** ‚Äî sample saved-exploration file with two H2 sections (durability angle grounded in REF-001's Limitations; safety-as-differentiation angle grounded in REF-001 + REF-002). The fixture README documents the manual smoke test for `/pilar:explore` and the deliberate non-validation of exploration files.

### Added (Phase 7)

- **`/pilar:scaffold-pillars`** ‚Äî proposes the ¬ß6.4 default 5-pillar set (unmet need; disease mechanism / pathophysiology; mechanism of action; clinical evidence; clinical value framework) tailored from `briefing.md`'s Strategic Priorities + Indication + Audiences + Competitive Context + Constraints + Evidence Generation Activities. Briefing-driven adjustments are explicitly enumerated (mechanism-tight indication ‚Üí consider merging mechanism pillars; safety-focused priority ‚Üí consider adding a tolerability-and-safety pillar; post-launch lifecycle ‚Üí consider adding an effectiveness-in-routine-practice pillar). On approval, generates `P-NN` ids in the user-confirmed order, computes kebab-case slugs (with user override), writes one `pillars/p-NN-<slug>.md` stub per approved pillar at `status: draft` (body H2 sections from the ¬ß7.5 schema with `_TBD_` placeholders pointing at the follow-up commands), and populates `roadmap.md`'s `## Pillars` table. Refuses to run if pillar files already exist (once-per-engagement). (commands/scaffold-pillars.md)
- **`/pilar:pillar-narrative <pillar-id>`** ‚Äî drafts the `## Strategic Rationale`, `## Narrative`, and `## Scope` sections of one pillar. Reads `briefing.md` (Strategic Priorities + Audiences + Indication + Competitive Context + Constraints), `knowledge-base/manifest.md` (REF entries grouped by `type` so the narrative is grounded in available evidence categories without naming specific REFs ‚Äî that's RS-level work), and any peer pillars at `status` ‚â• `narrative-approved` (each pillar's name + one-sentence narrative summary for cross-pillar consistency). Refuses if pillar is already at `narrative-approved` or higher (rewind sprint required per ¬ß5.3 to revise approved content). Continuation case (status still `draft` but body non-empty) is explicit ‚Äî user picks refine-or-discard. Edits the pillar file via Edit per section. Lexicon prompt (¬ß6.6) at end accumulates new entries (`<avoid> -> <preferred>`); status-transition prompt at end (yes/no/defer-until-sprint-close) sets `narrative-approved` on yes. Mode-specific commit messages. (commands/pillar-narrative.md)
- **`/pilar:pillar-statements <pillar-id>`** ‚Äî drafts SS-NN entries and nested RS-NN entries under `## Scientific Statements`. Refuses unless pillar is at `narrative-approved` (canonical case) or `statements-approved` (additive-with-warning case). Iteratively walks SS by SS, RS by RS ‚Äî for each RS, surfaces relevant manifest REF options based on the RS topic and *never invents REF ids that don't exist in the manifest*; if no manifest REF supports the proposed RS, flags this to the user (reshape, defer, or accept as orphan). Append-only IDs per `docs/CONVENTIONS.md` (`next_ss_n = max + 1` scoped to pillar; `next_rs_n = max + 1` scoped to parent SS). Writes via Edit (first-pass replaces the M1 `_TBD_` placeholder; additive case appends after the last existing SS without modifying prior content). Runs `scripts/detect-gaps.py` after drafting and offers inline `GAP-NNN` creation (mirroring `/pilar:ingest-kb` Step 12) for any orphans. Lexicon prompt; status-transition prompt; mode-specific commit messages. (commands/pillar-statements.md)
- **`examples/p7-fixtures/`** ‚Äî fully drafted P-04 Clinical Evidence pillar at `status: statements-approved` (2 SS, 4 RS) sourced cleanly against the P6 fixture manifest's REF-001..003. `roadmap-excerpt.md` shows the populated `## Pillars` table after `/pilar:scaffold-pillars`. README documents the manual smoke tests (`validate-schemas.py` and `detect-gaps.py` commands with expected outputs). Demonstrates the P7 exit criterion ("one pillar driven from scaffolding proposal through `statements-approved`") at the post-state. Following the qc-fixtures and p6-fixtures pattern, the fixture is *not* added to CI's `schema-validate` job (which scans `examples/fixtures/` only).

### Added (Phase 6)

- **`/pilar:ingest-kb`** ‚Äî auto-detected initial-intake or incremental ingestion of `knowledge-base/`. Initial mode proposes a taxonomy assignment against the five default subfolders seeded by `/pilar:init` (`clinical/`, `preclinical/`, `guidelines/`, `competitor/`, `other/`) with user-approved refinements including new subfolders; incremental mode diffs current files against the manifest's `file:` field set and appends only the delta. Manifest entries (per ¬ß7.4) are user-confirmed field-by-field on `citation`, `key_findings`, and `population` ‚Äî the Librarian never invents these from filenames alone. `REF-NNN` ids continue append-only per `docs/CONVENTIONS.md`. After the manifest commit lands, an orphan-RS scan runs automatically (Step 12 ‚Äî see below); a second separate commit captures any approved evidence-gap entries. Mode-specific user-facing commit messages (`chore(pilar): initial KB intake ‚Äî N sources catalogued` / `chore(pilar): KB intake ‚Äî N new source(s) added (M total)` / `chore(pilar): N evidence gap(s) opened from orphan-RS scan`). (commands/ingest-kb.md)
- **`/pilar:add-aspirational`** ‚Äî interactive walkthrough of every ¬ß7.7 aspirational-statement field for a given composite-id (`P-NN.SS-NN[.RS-NN]`). Validates the linked target exists in `pillars/`. The `evidence_generation_link` step reads `briefing.md`'s `## Evidence Generation Activities` section and presents the bullets as numbered choices plus a literal `candidate input` option (matching the schema's quoted literal for plans that don't exist yet). Generates the next `ASP-NNN` per `docs/CONVENTIONS.md` append-only rule, appends to `registers/aspirational-statements.md`, validates, proposes commit, waits for user approval. (commands/add-aspirational.md)
- **`scripts/detect-gaps.py`** ‚Äî orphan-RS predicate. Walks pillar files, parses SS/RS structure, and emits orphan candidates for any RS where the `sources:` line is missing entirely, the list is empty, or one or more cited `REF-NNN` ids do not exist in the manifest. Reuses `parse_frontmatter` / `section_body` / `find_headings` / `heading_id` / `ID_PATTERNS` from `scripts/validate-schemas.py` via `importlib.util` (the validator's hyphenated filename prevents a normal `import`, and a single source of truth for the markdown parsing primitives is preserved). Two output formats: markdown (default, human-friendly) and JSON (machine-friendly for slash-command consumption). Exit code 0 on success regardless of orphan count ‚Äî the script detects and emits, never gates.
- **`examples/p6-fixtures/`** ‚Äî self-contained P6 scenario for manual smoke testing. Three synthetic markdown sources (clinical / competitor / guideline) standing in for binary KB inputs per ¬ß3; post-ingest manifest with `REF-001..003`; a P-99 orphan pillar exercising both structural detection paths (empty `sources: []` and unresolved `REF-999`); post-detect `evidence-gaps.md` with two `GAP-NNN` entries; one `ASP-NNN` entry. README documents the expected `detect-gaps.py` output and the `validate-schemas.py` validation command.

### Added (Phase 5)

- **Real Fact-Checker subagent** (replacing the P2 walking-skeleton stub). `agents/fact-checker.md` upgraded to `tools: [Read]` with a path-based prompt template; performs real evaluation against cited sources, flagging overstatement, misattribution, source-strength mismatch, unsupported claims, and undefensible scientific-statement syntheses. Produces schema-conformant fact-check reports per ¬ß7.10.
- **Editor subagent** (`agents/editor.md`). Single file handling both `drafting` and `consolidated-draft` operating contexts via an `operating_context` flag. `tools: [Read, Edit]` ‚Äî applies meaning-preserving edits **surgically via the Edit tool** (one Edit call per change with sufficient `old_string` context for uniqueness). Permitted edits per ¬ß4.4/¬ß6.6 (lexicon swaps, ¬ß9 disallowed-pattern removal, tightening, cross-pillar harmonization in consolidated context); prohibited edits enumerated (factual claims, evidence linkages, hedging, frontmatter). Edit access scoped by directive to `{artifact_path}` only ‚Äî the lexicon and style guide remain read-only inputs. Output is just the schema-conformant editorial report (per ¬ß7.10) ‚Äî the artifact file is in its post-edit state by the time the subagent returns.
- **`/pilar:run-qc` Editor ‚Üí Fact-Checker sequencing.** Per ¬ß8 the Editor runs first; the parent inspects the Editor's surgical changes via `git diff <artifact>` and cross-checks the editorial report's claimed edit count against the diff (catches silently-failed Edit calls); the Fact-Checker then runs on the now-edited artifact. Discrete editorial commit at consolidated-draft context per ¬ß6.8. Sequencing enforced in command logic per decision #3.
- **`scripts/context-audit.py` registry pattern.** Refactored to `QC_HARNESSES` list of dicts; Editor harness registered with its own `allowed_vars` and `forbidden_tokens`. Per-harness `allowed_tools_sets` field replaces the global tools-list check: Fact-Checker constrained to `[set(), {"Read"}]`; Editor permitted `[{"Read"}, {"Read", "Edit"}]`. New `parse_tools_list` helper does proper YAML-list parsing. Adding the Strategic Reviewer at P8 is still a one-entry append.
- **`examples/qc-fixtures/`** ‚Äî self-contained QC test scenario (P-99 pillar with intentionally-bad RS, two-entry lexicon, style-guide with ¬ß9 defaults, kb-manifest, synthetic source content) for manual `/pilar:run-qc` smoke testing.

### Added (Phase 4)

- `/pilar:sprint-plan` ‚Äî opens a new sprint by drafting a plan from roadmap state and the prior sprint summary. Sprint-1 special case for briefing has pre-populated objectives. Writes `sprints/sprint-NN/plan.md`, updates roadmap (`current_sprint`, Active Workstreams), proposes commit message inline, waits for user approval. (commands/sprint-plan.md)
- `/pilar:sprint-close` ‚Äî closes the active sprint at the ¬ß5.3 checkpoint. Drafts the summary from work actually completed, presents at the checkpoint, executes the four-option state machine (Confirm / Request revisions [bounded or substantive] / Defer / Rewind) with branch-specific roadmap updates. Each terminal branch proposes its own commit message inline. (commands/sprint-close.md)
- `/pilar:sprint-amend` ‚Äî handles in-flight plan revisions per ¬ß5.2 ("an amended plan is proposed by the Primary Collaborator, approved by the user, and committed before further work proceeds"). Captures the delta in an `## Amendments` section appended to the plan. (commands/sprint-amend.md)
- `/pilar:init` extended to seed the full set of engagement-artifact stubs whose schemas now exist: `briefing.md`, `style-guide.md` (with the ¬ß9 disallowed-pattern defaults inherited verbatim from the schema), `lexicon.md`, `knowledge-base/manifest.md`, `registers/evidence-gaps.md`, `registers/aspirational-statements.md` ‚Äî alongside the existing `roadmap.md` seeding. All seeded stubs validate clean against `scripts/validate-schemas.py`.
- A static `CLAUDE.md` template dropped at the engagement repo root by `/pilar:init`. Triggers session resumption (¬ß5.4) on every Claude Code session start in the engagement directory: read roadmap ‚Üí most recent sprint summary ‚Üí active sprint plan ‚Üí orient the user briefly ‚Üí defer to user before offering work. Lists available `/pilar:*` slash commands.

## [0.1.0] ‚Äî 2026-05-02

First user-facing release. Locks the ¬ß7 artifact contract and ¬ß9 style defaults; establishes stable-ID conventions and validator enforcement.

### Added

- Full ¬ß7 schema coverage (13 schemas total): roadmap, briefing, sprint-plan, sprint-summary, kb-manifest, pillar, evidence-gaps, aspirational-statements, lexicon, style-guide, and the three QC reports (fact-check, editorial, strategic-alignment).
- ¬ß9 disallowed-pattern defaults baked into `schemas/style-guide.md`: em dashes for rhythm, antithetical "not X, it is Y" constructions, sentence-initial conjunctions, short punchy sentences for emphasis, hedging without scientific reason, marketing-register adjectives, first-person plural framing, rhetorical questions. Engagements inherit these defaults; clients can override per-pattern via the documented Overrides subsection.
- Stable-ID conventions (`REF-NNN`, `P-NN`, `SS-NN`, `RS-NN`, `GAP-NNN`, `ASP-NNN`, `FC-<sprint>-NNN`, `ED-<sprint>-NNN`, `CL-NNN`, `SA-<draft-tag>-NNN`) documented in [`docs/CONVENTIONS.md`](./docs/CONVENTIONS.md), with composite references like `P-NN.SS-NN.RS-NN` for cross-artifact linking.
- Schema validator (`scripts/validate-schemas.py`) extended with: per-artifact ID-format checks, uniqueness within document scope, nested SS/RS validation for pillar files, composite-reference validation for `linked_to` and `linked_statement` fields, and `sources: [REF-NNN, ...]` reference-format validation. Activated as the `schema-validate` CI job on every PR/push.
- Golden fixtures for all 13 schemas (`examples/fixtures/`) populated from the Aurelis Therapeutics / ALR-217 / r/r DLBCL synthetic engagement scenario defined in Appendix C of `IMPLEMENTATION_ROADMAP.md`.

### Pre-release context

The development arc to v0.1.0:

- **P1** (2026-05-01): Implementation roadmap committed; `LICENSE` (MIT, ¬© 2026 Clinton Avenue BioPharma Partners, LLC), `.gitignore`, and natural-language spec published. Repo public at https://github.com/JoshZiel83/pilar.
- **P2** (2026-05-02): Walking skeleton end-to-end. `.claude-plugin/plugin.json` and `.claude-plugin/marketplace.json` for marketplace install; `/pilar:init` scaffolds an engagement directory tree per ¬ß3 of the spec; stub Fact-Checker subagent (`agents/fact-checker.md` with `tools: []`) demonstrates the ¬ß4/¬ß8 Independence Contract; `/pilar:run-qc` parent harness with sentinel-bounded prompt template; static `scripts/context-audit.py` gates the harness against regression in CI.
- **P3** (2026-05-02): full schema contract + validator + first user-facing release. (This release.)

What pilar does NOT yet do at v0.1.0: real fact-checking (P5), real editorial enforcement (P5), KB ingestion (P6), pillar drafting workflow (P7), consolidation and strategic review (P8). The v1.0.0 cutover happens at P9 close per the implementation roadmap.
