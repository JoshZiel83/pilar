# Changelog

All notable changes to pilar are documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/), and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html). Per the project's load-bearing decision #5 (recorded in [`IMPLEMENTATION_ROADMAP.md`](./IMPLEMENTATION_ROADMAP.md)), explicit `version` is set in `.claude-plugin/plugin.json` only on tagged releases; the field is omitted between releases so the commit SHA serves as the version during active development.

## [Unreleased]

Phases 4, 5, 6, 7, and 8 closed: sprint engine + Primary Collaborator core (P4); Editor and Fact-Checker subagents with the Independence Contract codified for real evaluation (P5); KB Librarian with intake, gap detection, and aspirational-statement registration (P6); per-pillar development with scaffolding + narrative + statements + status state machine (P7); consolidation + Strategic Reviewer + whole-deliverable review + handoff + client-feedback re-entry pattern (P8). P7 refinement landed: bounded exploration command for source-content reads. Not yet tagged — release tagging is user-gated.

### Changed (breaking)

- **KB-manifest reference-id format changed from sequential `REF-NNN` to property-based** (closes part of issue #9, item 3). Manifest entry headings and `sources:` list values are now content-derived strings of the shape `^[A-Za-z0-9-]+(?:_[A-Za-z0-9-]+)+$` — recommended substructures (journal article, congress abstract, guideline / society report, pre-print) and the disambiguation rule are documented in [`docs/CONVENTIONS.md`](./docs/CONVENTIONS.md) §"KB manifest reference IDs". Rationale: the previous opaque sequential id required a manifest lookup at every citation site; the property form is self-documenting and makes likely duplicates collide on ingest. **Migration**: engagements created against earlier plugin versions remain on `REF-NNN` until manually migrated — rename each `### REF-NNN` heading and every `sources: [REF-NNN, ...]` list entry to the new form. The validator no longer accepts the legacy `^REF-\d{3}$` pattern. Surface area touched: `scripts/validate-schemas.py` (regex + docstring + error message); `docs/CONVENTIONS.md` (format-reference table, new grammar section, references in Where-they-are-referenced and Append-only sections); `schemas/kb-manifest.md` (placeholder comment); `scripts/detect-gaps.py` (docstring + reason text); slash-command and agent copy across `commands/{ingest-kb,research,explore,pillar-statements,pillar-narrative,run-qc,add-aspirational,init}.md` and `agents/fact-checker.md`; all four fixture corpora (`examples/{fixtures,p6-fixtures,p7-fixtures,p8-fixtures,qc-fixtures}`).

### Added

- **`/pilar:research` (issue #11)** — targeted PubMed and ClinicalTrials.gov search for the writer mid-drafting. Free-form `$ARGUMENTS` (natural-language instruction or specific query); Claude composes appropriate E-utility / `query.term` strings using briefing context, surfaces them, and lets the writer refine before running. Search and preview use the **PubMed** and **Clinical Trials** Claude.ai connectors (`mcp__claude_ai_PubMed__*`, `mcp__claude_ai_Clinical_Trials__*`) — both must be enabled. Default is small (5 hits per source) for the gap-fill use case; "show more" available on demand. Picks save as **provisional KB sources** under `knowledge-base/for_ingestion/`. Optional research note appends to `explorations/<pillar-id>.md` (mirrors `/pilar:explore`'s scratch-notes pattern). (commands/research.md)
- **`scripts/research-fetch.py`** — the canonical-bytes path for `/pilar:research`. Deterministic Python script (stdlib-only) that hits PubMed E-utilities (XML) and ClinicalTrials.gov v2 API (JSON) **directly**, bypassing the MCP layer. Same input + same calendar date → byte-identical output (only `fetched:` varies across days). Writes one markdown file per id with frontmatter (`provisional: true`, `source`, `pmid` or `nct_id`, structured fields) and verbatim abstract / brief summary content. The MCP-bypass is deliberate — MCP responses are LLM-mediated when Claude reads them, which reintroduces hallucination risk for any bytes that get cited later as fact. The script's `--verify <file>` mode re-fetches the source and diffs critical fields (title, sponsor / first author, journal, abstract / brief summary, whitespace-normalized) so drift can be detected at any time. Closes the user's load-bearing concern on issue #11 ("avoid a scenario in which Claude totally hallucinates an abstract or key details").
- **`knowledge-base/for_ingestion/`** — new staging subdirectory, scaffolded by `/pilar:init` alongside the existing five taxonomy subfolders. Convenience drop point for new sources awaiting ingestion: `/pilar:research` writes here, and writers may drop new PDFs / text files here manually. `/pilar:ingest-kb` continues to scan **all** of `knowledge-base/` (existing behavior preserved) — `for_ingestion/` is recommended, not required. During ingestion, files are moved out to the appropriate taxonomy subfolder, so `for_ingestion/` ends empty after each ingest run. `commands/init.md` updated to scaffold the directory and to refresh the CLAUDE.md `## Available pilar slash commands` section (which had become stale at v0.1.0 / Phase-2 wording).
- **`/pilar:ingest-kb` provisional-files branch** — Step 4 (taxonomy proposal) and Step 7 (entry proposal) now detect files with frontmatter `provisional: true` and treat them differently. Taxonomy is proposed using the structured metadata in the file's frontmatter (PubMed publication_type / MeSH terms; CT.gov lead_sponsor / conditions) instead of filename hints. Manifest entries pre-populate 6 of 7 §7.4 fields from the frontmatter; the user is asked only to confirm `key_findings` (the editorial summary, where hallucination guard does not apply because the writer must judge what is load-bearing) and to approve the batch. The non-provisional flow (manually-dropped sources) is unchanged. (commands/ingest-kb.md)
- **`status:` field on kb-manifest entries** — `schemas/kb-manifest.md` adds an optional `status: provisional | confirmed` line per REF entry. Default `confirmed` (absence allowed for backward compatibility — existing manifests do not specify and are treated as confirmed). `scripts/validate-schemas.py` validates the value when present (allowed values only). Provisional REFs flag sources whose file is metadata-only (PubMed abstract or CT.gov registration) awaiting upgrade when full text arrives. Upgrade flow is manual in v1: drop the full PDF in the appropriate `knowledge-base/<subfolder>/`, update the REF entry's `file:` and `status:` fields.
- **MCP prerequisites** — both the **PubMed** and **Clinical Trials** Anthropic-managed Claude.ai connectors must be enabled for `/pilar:research`. v1 documents the prerequisite; if either MCP is missing the command fails on first MCP call with a clear message. A WebFetch-against-the-upstream-APIs fallback for the search/preview arm is a v2 hardening if friction warrants — the script already hits both APIs directly, so the search/preview arm is the only fragile piece.

### Fixed

- **Issue #10 — engagement-runtime commands invoked scripts via bare `scripts/<x>.py` paths and failed with `FileNotFoundError` against engagement repos.** Per `scp-plugin-spec.md` §3, engagement repos are artifact containers — they do not (and should not) contain the plugin's runtime scripts. The convention `${CLAUDE_PLUGIN_ROOT}/scripts/<x>.py` was already in use by `/pilar:consolidate`; this change extends it to `/pilar:ingest-kb`, `/pilar:pillar-narrative`, `/pilar:pillar-statements`, `/pilar:add-aspirational`, `/pilar:scaffold-pillars`, and `/pilar:run-qc`. Two user-facing prose passages in `commands/ingest-kb.md` (and one in `commands/pillar-statements.md`) that suggested users run `python3 scripts/detect-gaps.py` ad-hoc from their own shell were rewritten to point at the slash-command paths that already wrap the script (`/pilar:ingest-kb`, `/pilar:pillar-statements`); since `${CLAUDE_PLUGIN_ROOT}` only expands inside Claude Code, there is no longer an out-of-band invocation form to advertise. Existing engagement repos pick up the fix automatically on the next plugin update — no engagement-repo backfill required.
- **`scripts/context-audit.py` extended with a slash-command script-path audit** (`audit_command_script_paths`). Scans every `commands/*.md` and fails if any line contains a bare `python3 scripts/<x>.py` (without the `${CLAUDE_PLUGIN_ROOT}/` prefix). Closes the regression vector that produced #10 — six of the seven script-invoking commands were written without the prefix and the convention drifted unnoticed across P5–P8.

### Added (Phase 8)

- **`scripts/consolidate.py`** — deterministic Python script that does the byte-level assembly of a consolidated draft per §6.7. Reads `briefing.md` + every approved pillar (`status: statements-approved` or `complete`) + `lexicon.md` + `style-guide.md`; demotes ATX headings (briefing H2 → H3; pillar H2 → H4 / SS H3 → H5 / RS H4 → H6; lexicon entries unchanged at H3; style-guide H2 → H3); drops pillar frontmatter and per-file H1; writes `consolidated/<cd-id>.md`. Auto-detects the next sequential `cd-NNN`; refuses if zero approved pillars; reuses `parse_frontmatter` from `validate-schemas.py` via the `importlib.util` pattern that `detect-gaps.py` established. Output bytes are determined by inputs — running twice on the same inputs produces byte-identical output (modulo the `created:` and "Assembled on" date stamps). Supports `--engagement-root`, `--draft-id`, `--output`, and `--dry-run` flags.
- **`/pilar:consolidate`** — thin slash-command wrapper around `scripts/consolidate.py`. Handles surrounding UX (engagement-state check, mtime-based drift warning vs the prior `cd-NNN` if any, orphan-RS audit via `scripts/detect-gaps.py`, validation, commit proposal); the script does the bytes deterministically. Open `GAP-NNN` and `ASP-NNN` register entries are deliberately excluded from the assembled body — registers are working-state, not deliverable content. (commands/consolidate.md)
- **`schemas/consolidated-draft.md`** — new artifact-type schema. Frontmatter-only (`artifact: consolidated-draft`, `draft_id`, `project`, `created`); body sections required by the validator: `## Briefing`, `## Pillars`, `## Lexicon`, `## Style Guide`. The consolidated draft is a derived view of the source pillars — the source pillars remain the canonical state of the engagement; the consolidated draft is regenerated from them on every consolidation.
- **`scripts/validate-schemas.py` extended** — new `cd` ID pattern (`^cd-\d{3}$`) added to `ID_PATTERNS`; new `FRONTMATTER_ID_RULES` table maps `(artifact, frontmatter-key) → ID_PATTERNS key` for foreign-key well-formedness checks on `consolidated-draft.draft_id` and `strategic-alignment-report.consolidated_draft`. (Closes the loop on the SA report's `consolidated_draft:` reference, which was previously not format-validated.)
- **Strategic Reviewer subagent** (`agents/strategic-reviewer.md`). `tools: [Read]`. Independence Contract per Appendix A: receives only briefing path + roadmap path + consolidated-draft path + `{operating_context}` + `{draft_id}`. Withholds drafting rationale, lexicon, style guide, KB manifest, source files, Editor's report, Fact-Checker's report, prior sprint summaries, per-pillar progress notes. Evaluates briefing strategic priorities against the deliverable: coverage, depth, pillar advancement, balance of emphasis across pillars, cross-pillar strategic contradictions. Produces strategic-alignment-report per §7.10 with finding-id format `SA-CD<N>-NNN`.
- **Editor subagent at consolidated-draft context is read-only** (`agents/editor.md`). Per the spec amendment to §4.4 / §6.8, the Editor at `operating_context: consolidated-draft` does not invoke Edit — it reads the consolidated draft and reports cross-pillar findings (terminology drift, lexicon adherence drift across pillars, claim duplication or contradiction, internal consistency at the consolidated level) as flagged items only. Findings are addressed by the writer editing the source pillars and re-consolidating, which is consistent with consolidation being deterministic. The Editor's drafting-context behavior is unchanged: still uses `Edit` surgically per the P5 design. The agent file declares `tools: [Read, Edit]` for the drafting use; the system prompt's per-context branch enforces no-edit at consolidated context.
- **`/pilar:run-qc --consolidated`** — extends `/pilar:run-qc` with a third sequential subagent step. The flag implies `--context consolidated-draft`, asserts the artifact's frontmatter is `artifact: consolidated-draft` (else stops with usage hint), and runs the Strategic Reviewer after the Editor (read-only) and Fact-Checker. SA report saved to `qc/strategic-alignment-reports/sprint-NN-strategic-cd-NNN.md`. At consolidated context the parent verifies `git diff` is empty after the Editor's pass as a contract check (the Editor must not modify the consolidated draft); there is no discrete editorial commit step. The §6.8 re-consolidation loop is surfaced when high-severity findings target the deliverable. (commands/run-qc.md)
- **`scripts/context-audit.py` Strategic Reviewer harness registered.** One-entry append to `QC_HARNESSES`: `allowed_vars` = `{operating_context, draft_id, briefing_path, roadmap_path, artifact_path}`; forbidden tokens enumerate the Appendix A withholds; `allowed_tools_sets` = `[{"Read"}]`. Audit now covers all three §4 QC subagents (Fact-Checker, Editor, Strategic Reviewer).
- **`/pilar:handoff`** — finalizes the engagement per §6.9 after the consolidated draft has cleared the WDR loop. Locates the latest `cd-NNN`, asserts the three reports exist, walks the user through user-confirmed clearance for every `severity: high` finding (reports carry no `status:` field — clearance is the user's judgment), updates `roadmap.md` (`## Status` to terminal narrative; appends a Decisions Log entry recording handoff date and `cd-NNN` delivered; Sprint History deliberately untouched — handoff is ceremonial after the final WDR sprint, not a sprint itself), proposes the final commit, then proposes a git tag (default `engagement-handoff-YYYY-MM-DD-cd-NNN`) with **explicit per-tag user authorization** per the saved release-gating policy applied analogously to engagement tags. Declining the tag prompt does not tag; the command prints the manual command for later use. (commands/handoff.md)
- **`/pilar:sprint-plan` post-handoff special case.** Second special-case branch alongside the existing Sprint-1 briefing branch: detects post-handoff state (roadmap `## Status` contains "Engagement complete" AND `consolidated/cd-NNN.md` exists) and proposes a templated client-feedback re-entry plan per §6.9 (reopen affected pillars named by the user; revise statements; re-consolidate to `cd-<NNN+1>`; re-run `/pilar:run-qc --consolidated`). Subsequent client rounds repeat this pattern. (commands/sprint-plan.md)
- **`/pilar:init` extended** — `consolidated/` directory added to the §3 directory tree alongside the existing dirs. Holds `consolidated/cd-NNN.md` whole-deliverable views; the directory is committed (with `.gitkeep`) so the first `/pilar:consolidate` invocation has a stable target. (commands/init.md)
- **Spec amendment** — `scp-plugin-spec.md` §4.4, §6.7, §6.8, §7.10, §8 amended (with user authorization — the user authored the spec) to record that consolidation is deterministic mechanical assembly and that all three QC roles are read-only at consolidated-draft checkpoints. The original spec language assumed Editor edits at consolidated stage with a structured change log and a discrete editorial commit; the amendment reflects the architectural shift that came out of P8 implementation review (deterministic consolidation surfaced the drift class created by editing the consolidated draft directly; pushing all corrective edits back to the source pillars eliminates that drift). The `CL-NNN` identifier (editorial change-log entry) is now obsolete and dropped from `docs/CONVENTIONS.md`; `schemas/editorial-report.md` no longer has a `## Change Log` section.
- **`examples/p8-fixtures/`** — self-contained P8 scenario for manual smoke testing. Three pillars at `status: statements-approved` (P-01 unmet need, P-04 clinical evidence reused from P7 fixture, P-05 clinical value framework); briefing with 3 strategic priorities; lexicon and style-guide; manifest with Smith_J_2024_Synth-J-Oncol..003. `consolidated/cd-001.md` is the deterministic mechanical assembly produced by `scripts/consolidate.py` (no embedded edits — the Editor is read-only at consolidated context). All three QC reports filed: editorial (4 ED flagged items naming pillar-level remediation in their `proposed_change` fields — three lexicon swaps and one antithetical-construction style observation), fact-check (1 medium + 2 low findings), strategic-alignment (1 high-severity finding engineered to exercise the §6.8 re-consolidation loop — Strategic Priority 2 under-addressed across the deliverable; the README documents the rerun expectation but does not ship `cd-002`). Following the qc-fixtures / p6-fixtures / p7-fixtures pattern, the fixture is *not* added to CI's `schema-validate` job (which scans `examples/fixtures/` only).

### P7 refinement

- **`/pilar:explore <pillar-id> [--angle "<hypothesis>"]`** — bounded, hypothesis-driven per-pillar exploration of KB source content. The writer brings a one-sentence angle; the command ranks manifest REFs by likely relevance, lets the user pick which to dig into, and uses the Read tool on the picked source files (PDFs included via the Read tool's `pages` parameter — defaulting to first 10 pages with user opt-in for deeper page ranges). Surfaces key passages and concepts; produces a 1–2 paragraph synthesis covering what the exploration supports, refutes, or suggests. Discarded by default; on save, appends a dated H2 section to `explorations/<pillar_id>.md` (one file per pillar; sections accumulate across invocations). Closes the original P7 gap where `/pilar:pillar-narrative` reads manifest entries grouped by `type` and `/pilar:pillar-statements` surfaces REF metadata, but neither command opens source files. (commands/explore.md)
- **`/pilar:init` extended** — `explorations/` directory added to the §3 directory tree alongside `pillars/`, `registers/`, `sprints/`, `qc/`, `knowledge-base/`. Holds optional per-pillar exploration notes; the directory is committed (with `.gitkeep`) but its files are scratch markdown — not §7 artifacts and not schema-validated. (commands/init.md)
- **`examples/p7-fixtures/explorations/p-04.md`** — sample saved-exploration file with two H2 sections (durability angle grounded in Smith_J_2024_Synth-J-Oncol's Limitations; safety-as-differentiation angle grounded in Smith_J_2024_Synth-J-Oncol + Doe_AB_2023_Synth-Lab-Med). The fixture README documents the manual smoke test for `/pilar:explore` and the deliberate non-validation of exploration files.

### Added (Phase 7)

- **`/pilar:scaffold-pillars`** — proposes the §6.4 default 5-pillar set (unmet need; disease mechanism / pathophysiology; mechanism of action; clinical evidence; clinical value framework) tailored from `briefing.md`'s Strategic Priorities + Indication + Audiences + Competitive Context + Constraints + Evidence Generation Activities. Briefing-driven adjustments are explicitly enumerated (mechanism-tight indication → consider merging mechanism pillars; safety-focused priority → consider adding a tolerability-and-safety pillar; post-launch lifecycle → consider adding an effectiveness-in-routine-practice pillar). On approval, generates `P-NN` ids in the user-confirmed order, computes kebab-case slugs (with user override), writes one `pillars/p-NN-<slug>.md` stub per approved pillar at `status: draft` (body H2 sections from the §7.5 schema with `_TBD_` placeholders pointing at the follow-up commands), and populates `roadmap.md`'s `## Pillars` table. Refuses to run if pillar files already exist (once-per-engagement). (commands/scaffold-pillars.md)
- **`/pilar:pillar-narrative <pillar-id>`** — drafts the `## Strategic Rationale`, `## Narrative`, and `## Scope` sections of one pillar. Reads `briefing.md` (Strategic Priorities + Audiences + Indication + Competitive Context + Constraints), `knowledge-base/manifest.md` (REF entries grouped by `type` so the narrative is grounded in available evidence categories without naming specific REFs — that's RS-level work), and any peer pillars at `status` ≥ `narrative-approved` (each pillar's name + one-sentence narrative summary for cross-pillar consistency). Refuses if pillar is already at `narrative-approved` or higher (rewind sprint required per §5.3 to revise approved content). Continuation case (status still `draft` but body non-empty) is explicit — user picks refine-or-discard. Edits the pillar file via Edit per section. Lexicon prompt (§6.6) at end accumulates new entries (`<avoid> -> <preferred>`); status-transition prompt at end (yes/no/defer-until-sprint-close) sets `narrative-approved` on yes. Mode-specific commit messages. (commands/pillar-narrative.md)
- **`/pilar:pillar-statements <pillar-id>`** — drafts SS-NN entries and nested RS-NN entries under `## Scientific Statements`. Refuses unless pillar is at `narrative-approved` (canonical case) or `statements-approved` (additive-with-warning case). Iteratively walks SS by SS, RS by RS — for each RS, surfaces relevant manifest REF options based on the RS topic and *never invents REF ids that don't exist in the manifest*; if no manifest REF supports the proposed RS, flags this to the user (reshape, defer, or accept as orphan). Append-only IDs per `docs/CONVENTIONS.md` (`next_ss_n = max + 1` scoped to pillar; `next_rs_n = max + 1` scoped to parent SS). Writes via Edit (first-pass replaces the M1 `_TBD_` placeholder; additive case appends after the last existing SS without modifying prior content). Runs `scripts/detect-gaps.py` after drafting and offers inline `GAP-NNN` creation (mirroring `/pilar:ingest-kb` Step 12) for any orphans. Lexicon prompt; status-transition prompt; mode-specific commit messages. (commands/pillar-statements.md)
- **`examples/p7-fixtures/`** — fully drafted P-04 Clinical Evidence pillar at `status: statements-approved` (2 SS, 4 RS) sourced cleanly against the P6 fixture manifest's Smith_J_2024_Synth-J-Oncol..003. `roadmap-excerpt.md` shows the populated `## Pillars` table after `/pilar:scaffold-pillars`. README documents the manual smoke tests (`validate-schemas.py` and `detect-gaps.py` commands with expected outputs). Demonstrates the P7 exit criterion ("one pillar driven from scaffolding proposal through `statements-approved`") at the post-state. Following the qc-fixtures and p6-fixtures pattern, the fixture is *not* added to CI's `schema-validate` job (which scans `examples/fixtures/` only).

### Added (Phase 6)

- **`/pilar:ingest-kb`** — auto-detected initial-intake or incremental ingestion of `knowledge-base/`. Initial mode proposes a taxonomy assignment against the five default subfolders seeded by `/pilar:init` (`clinical/`, `preclinical/`, `guidelines/`, `competitor/`, `other/`) with user-approved refinements including new subfolders; incremental mode diffs current files against the manifest's `file:` field set and appends only the delta. Manifest entries (per §7.4) are user-confirmed field-by-field on `citation`, `key_findings`, and `population` — the Librarian never invents these from filenames alone. `REF-NNN` ids continue append-only per `docs/CONVENTIONS.md`. After the manifest commit lands, an orphan-RS scan runs automatically (Step 12 — see below); a second separate commit captures any approved evidence-gap entries. Mode-specific user-facing commit messages (`chore(pilar): initial KB intake — N sources catalogued` / `chore(pilar): KB intake — N new source(s) added (M total)` / `chore(pilar): N evidence gap(s) opened from orphan-RS scan`). (commands/ingest-kb.md)
- **`/pilar:add-aspirational`** — interactive walkthrough of every §7.7 aspirational-statement field for a given composite-id (`P-NN.SS-NN[.RS-NN]`). Validates the linked target exists in `pillars/`. The `evidence_generation_link` step reads `briefing.md`'s `## Evidence Generation Activities` section and presents the bullets as numbered choices plus a literal `candidate input` option (matching the schema's quoted literal for plans that don't exist yet). Generates the next `ASP-NNN` per `docs/CONVENTIONS.md` append-only rule, appends to `registers/aspirational-statements.md`, validates, proposes commit, waits for user approval. (commands/add-aspirational.md)
- **`scripts/detect-gaps.py`** — orphan-RS predicate. Walks pillar files, parses SS/RS structure, and emits orphan candidates for any RS where the `sources:` line is missing entirely, the list is empty, or one or more cited `REF-NNN` ids do not exist in the manifest. Reuses `parse_frontmatter` / `section_body` / `find_headings` / `heading_id` / `ID_PATTERNS` from `scripts/validate-schemas.py` via `importlib.util` (the validator's hyphenated filename prevents a normal `import`, and a single source of truth for the markdown parsing primitives is preserved). Two output formats: markdown (default, human-friendly) and JSON (machine-friendly for slash-command consumption). Exit code 0 on success regardless of orphan count — the script detects and emits, never gates.
- **`examples/p6-fixtures/`** — self-contained P6 scenario for manual smoke testing. Three synthetic markdown sources (clinical / competitor / guideline) standing in for binary KB inputs per §3; post-ingest manifest with `Smith_J_2024_Synth-J-Oncol..003`; a P-99 orphan pillar exercising both structural detection paths (empty `sources: []` and unresolved `Missing_X_2099_Synth-Test`); post-detect `evidence-gaps.md` with two `GAP-NNN` entries; one `ASP-NNN` entry. README documents the expected `detect-gaps.py` output and the `validate-schemas.py` validation command.

### Added (Phase 5)

- **Real Fact-Checker subagent** (replacing the P2 walking-skeleton stub). `agents/fact-checker.md` upgraded to `tools: [Read]` with a path-based prompt template; performs real evaluation against cited sources, flagging overstatement, misattribution, source-strength mismatch, unsupported claims, and undefensible scientific-statement syntheses. Produces schema-conformant fact-check reports per §7.10.
- **Editor subagent** (`agents/editor.md`). Single file handling both `drafting` and `consolidated-draft` operating contexts via an `operating_context` flag. `tools: [Read, Edit]` — applies meaning-preserving edits **surgically via the Edit tool** (one Edit call per change with sufficient `old_string` context for uniqueness). Permitted edits per §4.4/§6.6 (lexicon swaps, §9 disallowed-pattern removal, tightening, cross-pillar harmonization in consolidated context); prohibited edits enumerated (factual claims, evidence linkages, hedging, frontmatter). Edit access scoped by directive to `{artifact_path}` only — the lexicon and style guide remain read-only inputs. Output is just the schema-conformant editorial report (per §7.10) — the artifact file is in its post-edit state by the time the subagent returns.
- **`/pilar:run-qc` Editor → Fact-Checker sequencing.** Per §8 the Editor runs first; the parent inspects the Editor's surgical changes via `git diff <artifact>` and cross-checks the editorial report's claimed edit count against the diff (catches silently-failed Edit calls); the Fact-Checker then runs on the now-edited artifact. Discrete editorial commit at consolidated-draft context per §6.8. Sequencing enforced in command logic per decision #3.
- **`scripts/context-audit.py` registry pattern.** Refactored to `QC_HARNESSES` list of dicts; Editor harness registered with its own `allowed_vars` and `forbidden_tokens`. Per-harness `allowed_tools_sets` field replaces the global tools-list check: Fact-Checker constrained to `[set(), {"Read"}]`; Editor permitted `[{"Read"}, {"Read", "Edit"}]`. New `parse_tools_list` helper does proper YAML-list parsing. Adding the Strategic Reviewer at P8 is still a one-entry append.
- **`examples/qc-fixtures/`** — self-contained QC test scenario (P-99 pillar with intentionally-bad RS, two-entry lexicon, style-guide with §9 defaults, kb-manifest, synthetic source content) for manual `/pilar:run-qc` smoke testing.

### Added (Phase 4)

- `/pilar:sprint-plan` — opens a new sprint by drafting a plan from roadmap state and the prior sprint summary. Sprint-1 special case for briefing has pre-populated objectives. Writes `sprints/sprint-NN/plan.md`, updates roadmap (`current_sprint`, Active Workstreams), proposes commit message inline, waits for user approval. (commands/sprint-plan.md)
- `/pilar:sprint-close` — closes the active sprint at the §5.3 checkpoint. Drafts the summary from work actually completed, presents at the checkpoint, executes the four-option state machine (Confirm / Request revisions [bounded or substantive] / Defer / Rewind) with branch-specific roadmap updates.
